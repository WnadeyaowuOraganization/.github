# 编程测试整合方案 - 技术整合计划

## 一、背景

### 1.1 现状痛点
- CI反馈慢：20-35分钟
- 无法本地验证：编程CC需等待CI结果
- 环境干扰：多CC共享dev环境

### 1.2 核心诉求
**编译环境与测试环境共用**，避免重复构建

## 二、方案整合

### 2.1 排程经理设计：共享Volume方案

**核心组件**：
```
/home/ubuntu/.m2/repository        # 共享Maven依赖
/home/ubuntu/.pnpm-store           # 共享pnpm缓存
/apps/wande-ai-backend-kimi{N}/    # 各kimi独立产物
/apps/wande-ai-front-kimi{N}/      # 各kimi独立前端产物
```

**关键技术点**：
- 只读共享 + 本地写入（避免并发冲突）
- 文件锁机制（后台更新时使用）
- 环境变量注入（run-cc.sh配置）

### 2.2 研发经理设计：编译-测试一体化

**核心改进**：
- 合并build和e2e-test为一个job
- 消除重复启动等待（-120秒）
- PID持久化 + 进程监控 + 优雅关闭

**4阶段流程**：
1. 编译（如需要）
2. 启动服务
3. 执行测试
4. 清理资源

### 2.3 整合方案

**架构图**：
```
┌─────────────────────────────────────────────────────────────┐
│                    编程CC本地环境                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐     │
│  │  编译阶段   │───→│  启动阶段   │───→│  测试阶段   │     │
│  │             │    │             │    │             │     │
│  │ - mvn package│   │ - nohup    │    │ - playwright│     │
│  │ - pnpm build│    │ - 健康检查  │    │ - 结果收集  │     │
│  │             │    │             │    │             │     │
│  └──────┬──────┘    └──────┬──────┘    └──────┬──────┘     │
│         │                  │                  │            │
│         └──────────────────┼──────────────────┘            │
│                            │                               │
│                   ┌────────▼────────┐                      │
│                   │   共享Volume    │                      │
│                   │                 │                      │
│                   │ ~/.m2/repository│                      │
│                   │ ~/.pnpm-store   │                      │
│                   │ /apps/...       │                      │
│                   └─────────────────┘                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**整合关键点**：

| 设计 | 排程经理贡献 | 研发经理贡献 | 整合方式 |
|------|-------------|-------------|---------|
| 依赖缓存 | 共享Maven/pnpm目录 | 环境变量注入 | run-cc.sh统一配置 |
| 产物复用 | 各kimi独立产物目录 | 同job内直接使用 | 产物路径标准化 |
| 服务生命周期 | - | PID持久化+优雅关闭 | 复用研发经理设计 |
| 健康检查 | - | 20轮×3秒→5轮×1秒 | 复用研发经理设计 |

## 四、预期效果

| 指标 | 当前 | 优化后 | 改善 |
|------|------|--------|------|
| 编译时间 | 5-8分钟 | 2-3分钟 | -60% |
| 启动等待 | 60-120秒 | 0秒 | -100% |
| 测试时间 | 10-15分钟 | 6-9分钟 | -40% |
| **总计** | **20-35分钟** | **8-15分钟** | **-50%~60%** |

## 五、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| I/O竞争 | 20个CC同时读取共享目录 | 监控iostat，必要时错开启动 |
| 内存不足 | 32GB < 60GB峰值 | 配置Swap，限制并发编译数 |
| 缓存失效 | 代码更新后缓存未刷新 | 版本号变化自动失效 |
| 权限问题 | 无法写入共享目录 | 统一使用ubuntu用户 |
