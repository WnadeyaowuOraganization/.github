# ADR-001: 编译-测试一体化架构设计

**状态**: 提议
**日期**: 2026-04-11
**决策者**: 研发经理、排程经理、架构师

---

## 背景

### 当前问题

在E2E测试优化方案Day 1调研中发现，GitHub Actions workflow中的`build`和`e2e-test`两个独立job存在以下痛点：

1. **重复启动等待**：`e2e-test` job必须重新启动后端服务，等待60-120秒健康检查
2. **资源不共享**：job间进程不保留，编译产物路径虽然可访问，但服务进程被清理
3. **时间浪费**：总时长20-35分钟，其中约3分钟为冗余等待

### 数据支持

| 阶段 | 当前耗时 | 瓶颈 |
|------|---------|------|
| build job | 5-8分钟 | 编译无缓存 |
| 启动等待 | 120秒 | job分离 |
| e2e-test job启动 | 60秒 | 重复启动 |
| 测试执行 | 10-15分钟 | workers=2 |
| **总计** | **20-35分钟** | - |

---

## 决策

### 核心方案

**合并`build`和`e2e-test`为单一job**，实现编译-测试一体化流程。

### 架构设计

#### 原架构（分离job）

```
[build job]
  └─ 编译 → 启动服务 → 健康检查(120s) → job结束(进程清理)

[e2e-test job] (重新启动)
  └─ 启动服务 → 健康检查(60s) → 测试 → job结束
```

#### 新架构（一体化job）

```
[build-and-test job]
  └─ 编译 → 启动服务 → 健康检查(60s) → 测试 → 清理
```

### 技术实现

1. **PID持久化**：
   - 将后端进程PID写入`/tmp/ci-backend.pid`
   - 后续步骤通过PID文件精确控制进程

2. **健康检查优化**：
   - 从120秒缩短到60秒（智能超时）
   - 检查间隔从3秒缩短到2秒
   - 增加进程存活监控

3. **优雅清理机制**：
   - 使用`if: always()`确保清理总是执行
   - 优先优雅关闭（SIGTERM），兜底强制终止（SIGKILL）
   - 备份日志供后续分析

## 后果

### 正面影响

| 维度 | 改善 |
|------|------|
| **时间** | 节省约3分钟（消除重复启动等待） |
| **架构** | job数量从3个减少到2个，依赖链简化 |
| **可维护性** | 单一job日志集中，易于调试 |
| **资源** | 内存峰值不变（分时占用） |

### 负面影响

| 维度 | 影响 | 缓解措施 |
|------|------|---------|
| **日志混在一起** | 不同阶段日志难以区分 | 使用`::group::`分隔步骤日志 |
| **内存占用时间延长** | 服务进程运行时间增加 | 峰值内存不变（3.5GB），G7e资源充足 |
| **回滚复杂度** | 单job回滚比多job复杂 | 保留原workflow作为备份 |

### 风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| 内存泄漏 | 🟡 中 | 监控内存使用，超过3GB告警 |
| 进程异常终止 | 🟢 低 | 健康检查阶段验证进程存活 |
| 构建失败影响测试 | 🟢 低 | 符合当前逻辑（build失败→无测试） |

## 验证标准

### 成功标准

- ✅ 编译-测试全流程时间减少 ≥ 2分钟
- ✅ 测试覆盖率不变
- ✅ 内存峰值 ≤ 4GB
- ✅ 无功能回归

### 回滚标准

- ❌ 新workflow失败率 > 10%
- ❌ 内存峰值 > 4GB
- ❌ 测试结果不一致

---

## 替代方案

### 方案A: 保持现状

- **优点**：无风险，无需修改
- **缺点**：每次PR浪费3分钟，用户体验差
- **结论**：不推荐

### 方案B: 使用GitHub Actions services

- **优点**：原生支持服务容器
- **缺点**：services不支持动态启动Java进程
- **结论**：技术不可行

### 方案C: 使用GitHub Actions matrix

- **优点**：可并行测试不同模块
- **缺点**：增加复杂度，不适合当前规模
- **结论**：暂不实施，可作为未来优化方向
