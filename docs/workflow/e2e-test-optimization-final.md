## 最终整合方案: 渐进式混合模式

### 核心原则

**资源不是瓶颈,开发和运维成本才是**。因此采用分级隔离策略,而非全量独立环境。

---

### 方案架构

#### L0: 共享dev环境 (默认, 0成本)

**适用场景**:
- 简单CRUD (单表修改)
- 文档/配置变更
- 样式调整
- 纯后端或纯前端修改

**实施方式**:
- 编程CC在feature分支开发
- 提交前运行单元测试 + 编译检查
- PR创建后CI自动E2E测试
- **无额外基础设施**

**优势**:
- 零成本,零运维
- 符合现有工作流
- CI已有完善反馈机制

---

#### L1: 测试环境池 (按需分配, 低成本)

**适用场景**:
- 中等复杂度Issue (2-3张表)
- 前后端联动修改
- 需要隔离测试但不需要完全独立

**实施方式**:
- 维护**5个测试环境池** (非20个)
- 动态分配: CC启动时申请 → 使用 → 释放
- 资源消耗: 约7.5GB (vs 方案A的30GB)
- 端口范围: backend:7101-7105 / frontend:8101-8105 / db:5401-5405

**环境池化脚本**:
```bash
# 新增: scripts/cc-test-pool.sh
# 功能: 管理5个测试环境的分配和释放

# 申请环境 (返回环境编号 1-5)
ENV_ID=$(scripts/cc-test-pool.sh acquire kimi1)

# 释放环境
scripts/cc-test-pool.sh release $ENV_ID
```

**调度策略**:
- 研发经理CC在run-cc.sh中增加 `--isolation=L1` 参数
- 环境池满时排队等待 (最大等待5分钟)
- 使用超时(2小时)自动释放

**优势**:
- 75%资源节省 (vs 方案A)
- 运维成本可控 (5个环境 vs 20个)
- 满足大部分隔离需求

---

#### L2: 完全独立环境 (极少数场景, 高成本)

**适用场景**:
- 复杂Issue (5+张表,跨模块)
- 架构级重构
- 数据库Schema变更
- 性能压测

**实施方式**:
- 按需创建,使用后销毁
- 端口动态分配 (避免冲突)
- 资源消耗: 约1.5GB/环境

**触发条件** (由研发经理CC判断):
```bash
run-cc.sh --issue 1234 --module backend --dir kimi1 --effort high --isolation=L2
```

**限制**:
- 同时最多2个L2环境运行
- 使用超时(4小时)自动清理
- 需要研发经理CC明确指定

**优势**:
- 完全隔离,无干扰
- 按需创建,不常驻

## 关键决策

### 为什么不采用方案A (每个CC独立环境)?

**排程经理视角**:
1. **运维成本激增**: 20个独立PostgreSQL实例,端口管理复杂度指数级上升
2. **数据库版本管理困难**: 每个环境独立Flyway migration,难以保证一致性
3. **资源浪费**: 大部分时间环境空闲,占用30GB内存

**架构师视角**:
1. **技术风险高**: 80个进程并发,系统稳定性风险
2. **回滚复杂**: 出问题时需要回滚20个环境,恢复时间长
3. **维护负担**: 监控、日志、调试成本激增

**共识**: 资源虽然充足,但运维成本和风险不可接受。

---

### 为什么选择混合模式?

**研发经理视角**:
1. **快速反馈优先**: 简单Issue (L0) 无需等待,复杂Issue (L1/L2) 按需隔离
2. **分级策略灵活**: 根据Issue复杂度动态选择,不浪费资源
3. **开发效率提升**: 避免方案B的"一刀切"延迟

**排程经理视角**:
1. **资源成本可控**: 大部分Issue走L0 (零成本), 少数走L1 (低成本), 极少走L2
2. **运维负担低**: 5个池化环境 vs 20个独立环境
3. **风险可控**: 分级隔离,出问题影响范围小

**架构师视角**:
1. **技术实现可行**: 环境池化是成熟模式,风险低
2. **可扩展性好**: 未来可根据实际使用调整池大小
3. **系统稳定性**: 渐进式推广,每一步都可以回滚

**共识**: 混合模式平衡了效率、成本、风险三者。

## 总结

**最终方案 = 渐进式混合模式**:
- L0 (共享dev): 默认选择,零成本
- L1 (环境池): 按需隔离,低成本
- L2 (完全独立): 极少数场景,高成本

**核心优势**:
1. ✅ 平衡效率、成本、风险
2. ✅ 资源消耗可控 (0-7.5GB)
3. ✅ 运维负担低 (5个池化环境)
4. ✅ 分级策略灵活,按需扩展
5. ✅ 渐进式推广,每一步可回滚

**不采用方案A的原因**:
- ❌ 资源浪费 (30GB)
- ❌ 运维成本高 (20个独立环境)
- ❌ 风险不可控 (80个进程并发)
- ❌ 数据库管理复杂
