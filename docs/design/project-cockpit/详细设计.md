## 1. 页面概述

- **路由**: `/wande/project/cockpit`
- **菜单层级**: 项目中心 > 项目全景控制表
- **权限标识**: `wande:cockpit:*`
- **所属模块**: `module:fullstack`
- **核心视图**: `v_cockpit_zone_metrics`（聚合视图，JOIN budget_zone + plan_milestone + bom_items + actuals）
- **原型架构**: 单页 1 Tab（`cockpit-overview.html`）— 单文件架构
- **依赖服务**:
  - 预算模块（`budget_zone` / `budget_item`，提供区域预算数据）
  - 计划模块（`project_plan` / `plan_milestone`，提供里程碑进度数据）
  - BOM 模块（`bom_items`，提供物料交付数据）
  - Redis 缓存（`cockpit:project:{id}` TTL=5min，计划/采购/BOM 任一变更触发 cache evict）
  - EVM 计算（服务层聚合：PV 来自里程碑、EV 来自已交付任务工时、AC 来自 `budget_item.consumed_amount`）

### 1.1 业务背景

万德在建项目数量超过 10 个，吴耀需要在每天早会前 10 分钟快速了解全部项目的健康状态。当前信息分散在预算系统、计划甘特图、BOM 清单三个模块，需要逐一切换查看，会前准备时间超过 30 分钟。

项目全景控制表的目标是：

1. **「一张表」看全局**：将区域×预算×计划×BOM×健康度五个维度集中在一张矩阵中，5 秒内定位问题区域。
2. **EVM 标准化量化**：引入挣值管理（EVM）指标（SPI/CPI）为项目整体健康度提供客观数字依据，而非主观判断。
3. **多项目横向对比**：同一视图对比多个项目的关键指标，识别共性风险（如全部项目的设备安装阶段都在延期）。
4. **EVM 趋势预警**：SPI 或 CPI 连续 3 周下降时，系统自动标记该项目为「趋势恶化」并推送告警。

### 1.2 用户角色

| 角色 | 典型操作 | 权限范围 |
|------|---------|---------|
| 吴耀（超管） | 每日查看全景控制表 / 分析 EVM 趋势 / 下钻异常区域 | 全量只读 |
| 总监 | 查看本负责项目的全景视图 / 接收趋势告警 | 本负责项目只读 |
| 项目经理 | 查看本项目全景视图 | 本项目只读 |
| 财务专员 | 查看 EVM 和预算消耗 | 财务维度只读 |

### 2.1 顶部项目选择栏

**功能描述**: 用户在此选择要查看的项目（或多个项目对比），设置时间区间，触发全页数据刷新。顶部固定不随页面滚动。

- **组件**:
  - `ProjectSelector`（项目多选下拉，支持全选「当前在建项目」快捷按钮，支持关键字搜索，最多同时选 5 个项目对比）
  - `DateRangePicker`（时间区间选择：EVM 计算基准期间，默认为项目开始日期至今）
  - `RefreshBtn`（手动刷新按钮，带「最后更新时间」提示：`缓存于 N 分钟前`）
  - `ExportBtn`（导出当前视图为 Excel 或 PDF 报告）
  - `SettingBtn`（视图设置：选择显示哪些区域/科目/维度）

- **数据源**: 项目选择后联动触发所有其他区域的数据请求（通过 Pinia store `setCockpitProject` action 广播）

- **交互**:
  - 项目下拉默认选中用户「最近查看的项目」（localStorage 记忆），首次访问默认选第一个在建项目
  - 切换项目 → 全页所有区域（§2.2-§2.7）同步重新请求数据，Loading 遮罩覆盖各区域
  - 多项目模式（选 2-5 个）→ §2.4（健康度雷达）和 §2.6（多项目对比卡片）切换为多项目对比视图
  - 时间区间调整 → 仅 §2.7（EVM 趋势曲线）重新请求，其他区域不刷新
  - 「缓存于 N 分钟前」标签：若 N ≥ 5，标签变橙色并出现「数据可能不是最新」提示

### 2.2 EVM 全局仪表盘（5 KPI）

**功能描述**: 以 5 张数字卡片展示当前选中项目的 EVM 核心指标：PV（计划价值）/ EV（挣值）/ AC（实际成本）/ SPI（进度绩效指数）/ CPI（成本绩效指数）。让吴耀在 5 秒内判断项目整体健康状态。

- **组件**:
  - `EvmKpiCard`（单张 KPI 卡：指标名 / 当前值 / 趋势箭头（环比上周）/ 颜色状态（绿/橙/红））
  - `EvmFormulaTooltip`（悬停 EVM 公式说明：PV=计划里程碑价值 / EV=已交付任务价值 / AC=实际消费金额）
  - `EvmStatusBadge`（项目整体状态 badge：HEALTHY / AT_RISK / CRITICAL，右上角固定显示）
  - `EvmTrendMiniChart`（每张 KPI 卡底部迷你折线图：近 8 周该指标的走势 sparkline）

- **数据源**:
  - `GET /wande/cockpit/evm`（EVM 指标：`projectId` + `asOfDate`，返回 PV / EV / AC / SPI / CPI + 趋势数据）

- **交互**:
  - 5 张 KPI 卡横向并排，响应式布局（PC 5列 / 平板 3 列）
  - SPI < 0.9（进度落后）→ SPI 卡背景橙色；SPI < 0.8 → 红色
  - CPI < 0.9（超支）→ CPI 卡背景橙色；CPI < 0.8 → 红色
  - 悬停 KPI 卡 → 展示 EVM 公式说明（PV 来自哪些里程碑，EV 如何计算，AC 来自哪个表）
  - 迷你折线图显示近 8 周趋势，下降趋势用红色线，上升用绿色
  - 连续 3 周 SPI 或 CPI 下降 → `EvmStatusBadge` 自动变为 `CRITICAL`，触发企微告警（通过后端定时任务）

- **代码现状**: 🔴 需新建
  - EVM 计算逻辑新建：`EVM_PV` = JOIN `plan_milestone.plan_value`（根据计划日期权重分配）；`EVM_EV` = JOIN 已完成任务的计划价值；`EVM_AC` = JOIN `budget_item.consumed_amount`
  - SPI = EV/PV，CPI = EV/AC，Service 层 BigDecimal 计算，精度保留 4 位

- **已有文件**: 无

- **三级边界**:

```
✅ Always（必须做）:
- EVM 计算全程使用 BigDecimal，不使用 double（SPI/CPI 展示精度 2 位小数，内部计算 4 位）
- PV 计算基于 project_plan 里程碑的计划完成日期（非任务级），EV 基于已实际完成的任务工时，AC 基于 budget_item.consumed_amount，三者口径不可混用
- SPI / CPI 颜色阈值（0.9 橙 / 0.8 红）存 budget_param 表，超管可调整不需发版

⚠️ Ask First（需确认再做）:
- PV 按里程碑权重分配（当前按里程碑数量平均分配计划总值），若改为按工期加权需评估历史数据重算成本
- 增加 EAC（完工估算）/ VAC（完工偏差）两个衍生指标
- EVM 告警（连续 3 周下降）的通知范围（当前为项目负责人+总监，是否需要加吴总直接通知）

🚫 Never（禁止做）:
- 不得在前端计算 SPI / CPI（EVM 指标计算在 Service 层，防止因数据精度导致前后端不一致）
- 不得将 PV/EV/AC 的计算口径在不同接口中混用（§3.2 /cockpit/evm 接口是唯一计算入口）
```

### 2.3 三维矩阵主表（区域×预算×计划×BOM×健康度）

**功能描述**: 以「区域」为行，以「预算执行 / 计划进度 / BOM 交付 / 健康度」为列，构成可折叠的核心矩阵。吴耀在此一眼扫描哪个区域存在异常，哪个维度拖后腿。

- **组件**:
  - `CockpitMatrix`（主矩阵表格：冻结首列「区域名」，横向 4 个维度分组列）
  - `MatrixRow`（单区域行：含区域编码 / 名称 / 4 个维度的进度条 + 数值 + 状态颜色）
  - `MatrixDimensionHeader`（列分组头：「预算」「计划」「BOM」「健康度」4 个分组，可整组折叠/展开）
  - `RowDrilldownBtn`（行末「下钻」按钮，点击展开该区域的明细子表格）
  - `ZoneDetailSubTable`（区域明细子表：该区域各科目预算执行详情 + 里程碑列表 + BOM 物料到货状态）
  - `MatrixLegend`（颜色图例：绿色=正常 / 橙色=需关注（60-80%）/ 红色=危险（>80%）/ 灰色=未开始）

- **数据源**:
  - `GET /wande/cockpit/aggregate`（三维矩阵聚合数据，`projectId`，返回所有区域的多维度汇总）

- **交互**:
  - 矩阵行按「健康度评分」升序排列（问题区域排最前）
  - 每个维度列的单元格包含：迷你进度条（0-100%）+ 数值文本 + 颜色状态
  - 点击某行「下钻」→ 行下方展开子表，子表包含：该区域各科目预算明细 + 该区域里程碑（计划/实际日期）+ 该区域 BOM 物料到货状态
  - 「预算」列分组折叠 → 该分组所有列隐藏，行内显示「▶」折叠标志，节省横向空间
  - 矩阵支持关键字搜索（筛选区域名），匹配区域高亮，不匹配行隐藏
  - 矩阵列头点击 → 按该列排序（如按「预算使用率」降序，快速找到超支最严重的区域）

- **代码现状**: 🔴 需新建
  - `v_cockpit_zone_metrics` 数据库聚合视图需新建（JOIN 4 个表/视图）
  - 聚合查询可能耗时较长（10+ 表 JOIN），必须走 Redis 缓存（TTL=5min）
  - 健康度评分算法：预算分（40%，基于使用率偏离度）+ 计划分（40%，基于里程碑完成情况）+ BOM 分（20%，基于物料到货率）

- **已有文件**: 无

- **三级边界**:

```
✅ Always（必须做）:
- 矩阵聚合接口必须走 Redis 缓存（TTL=5min），任何预算/计划/BOM 数据变更需发布 CockpitCacheEvictEvent 触发主动失效
- 矩阵行数超过 50 行时开启前端虚拟滚动（避免 DOM 节点过多卡顿）
- 下钻子表数据通过单独接口按需加载（§3.1 /cockpit/aggregate 不包含明细，避免首屏数据量过大）

⚠️ Ask First（需确认再做）:
- 新增第 5 个维度列（如「安全」维度，基于安全事故记录）
- 区域行按用户自定义顺序排列（当前固定按健康度评分升序）
- 矩阵支持列宽拖拽调整（前端复杂度较高，当前固定列宽）

🚫 Never（禁止做）:
- 不得在矩阵主接口中返回区域明细数据（明细仅在下钻时按需加载，防止首次加载响应过大）
- 不得允许非本项目用户看到其他项目的区域数据（Service 层根据 JWT 用户权限过滤 projectId）
- 不得在前端计算健康度评分（评分在 Service 层统一计算，防止前后端评分不一致导致颜色混乱）
```

### 2.4 健康度五维雷达图

**功能描述**: 以雷达图（蜘蛛网图）可视化展示项目的五维健康度评分：预算合规度 / 计划准时度 / BOM 交付率 / 供应商准时率 / 变更控制度。多项目模式下在同一雷达图叠加对比，快速识别各项目的短板维度。

- **组件**:
  - `HealthRadarChart`（ECharts 雷达图：5 个坐标轴，0-100 分值域，多项目叠加时每个项目一条线，颜色区分）
  - `RadarDimensionCard`（雷达图右侧分解卡：5 个维度的当前分 + 简短说明 + 上周对比）
  - `RadarLegend`（多项目模式下的图例：项目名 + 颜色对应关系）
  - `HealthScoreGauge`（雷达图中央的综合健康度仪表盘：0-100 分的半圆仪表，颜色对应 HEALTHY/WARNING/DANGER）
  - `RadarHistoryBtn`（「查看历史雷达」按钮，展示近 8 周雷达图动画演变）

- **数据源**:
  - `GET /wande/cockpit/health-radar`（雷达 5 维度评分，`projectId`（可多个），返回各项目各维度分值）

- **交互**:
  - 单项目模式：雷达图展示一个项目 5 维度评分，中央仪表显示综合分（加权平均）
  - 多项目模式（选了 2-5 个项目）：雷达图叠加显示所有项目，hover 某条线显示该项目信息
  - 点击雷达图某个顶点（如「BOM 交付率」）→ 展开该维度的详情卡（BOM 到货率分区域明细）
  - 「查看历史雷达」→ 播放近 8 周雷达图动画（每周快照），直观看到评分变化趋势
  - 综合分 < 60 → 仪表盘区域红色，页面顶部出现红色预警横幅「项目健康度严重不足」
  - 雷达图 hover 顶点显示 tooltip：维度名 / 当前分 / 上周分 / 行业基准分（参考值，来自 budget_baseline）

  - ECharts 雷达图组件封装为 `HealthRadarChart.vue`

- **已有文件**: 无

- **三级边界**:

```
- 雷达图 5 个维度的评分算法系数（权重）存配置表（budget_param），超管可调整，无需发版
```

### 2.5 异常区域排行

**功能描述**: 汇总当前项目中健康度最差的区域（TOP-5 问题区域），以榜单形式展示，每个区域显示主要问题描述（自动生成的问题摘要：「预算超支 15%」「里程碑延期 8 天」「3 个物料未到货」）。

- **组件**:
  - `ExceptionRankList`（问题区域榜单：排名 + 区域名 + 健康分 + 主要问题标签 + 「详情」跳转链接）
  - `ExceptionTagGroup`（问题标签组：每个区域最多 3 个自动生成的问题标签，按严重程度排序）
  - `ExceptionDetailBtn`（「处理问题」按钮：跳转到对应模块的具体操作页面）
  - `RefreshRankBtn`（单独刷新排行按钮，不触发全页刷新）

- **数据源**:
  - `GET /wande/cockpit/aggregate`（复用三维矩阵主接口，响应中包含 `topExceptionZones` 字段）

- **交互**:
  - 榜单默认展示健康度 < 70 分的区域，按健康分升序（最差的排第 1）
  - 健康分 < 50 的区域排名前带红色火焰图标 🔴，50-70 带橙色警告图标 🟡
  - 问题标签自动生成规则：
    - 预算使用率 ≥ 90%：「预算消耗 {rate}%，剩余不足」
    - 里程碑延期 ≥ 3 天：「里程碑延期 {days} 天」
    - BOM 到货率 < 80%：「{count} 个物料未到货」
  - 点击「处理问题」→ 跳转到对应模块：预算超支 → `/wande/budget#zone={zoneId}`；计划延期 → `/wande/project/plan#task={taskId}`；BOM 未到货 → `/wande/bom#zone={zoneId}`
  - 若所有区域健康度 ≥ 80 → 榜单区域显示「✅ 所有区域运行正常，无异常告警」绿色提示

- **代码现状**: 🔴 需新建（问题摘要自动生成逻辑需在聚合接口中实现）

- **已有文件**: 无

- **三级边界**:

### 4.1 数据库聚合视图 v_cockpit_zone_metrics

**对应 Flyway**: `V20260427_21__create_cockpit_zone_metrics_view.sql`

```sql
-- 项目区域多维度指标聚合视图（供 cockpit/aggregate 接口使用）
CREATE OR REPLACE VIEW `v_cockpit_zone_metrics` AS
SELECT
  bz.id                             AS zone_id,
  bz.zone_code,
  bz.zone_name,
  bz.zone_type,
  bz.project_id,
  -- 预算维度
  COALESCE(SUM(bi.total_amount), 0)    AS budget_total,
  COALESCE(SUM(bi.consumed_amount), 0) AS budget_consumed,
  CASE
    WHEN COALESCE(SUM(bi.total_amount), 0) = 0 THEN 0
    ELSE ROUND(COALESCE(SUM(bi.consumed_amount), 0) / SUM(bi.total_amount), 4)
  END                                  AS budget_usage_rate,
  -- 计划维度（里程碑）
  COUNT(pm.id)                         AS milestone_total,
  COUNT(CASE WHEN pm.status = 'COMPLETED' THEN 1 END) AS milestone_completed,
  COUNT(CASE WHEN pm.status = 'DELAYED' THEN 1 END)   AS milestone_delayed,
  MAX(CASE WHEN pm.status = 'DELAYED'
      THEN DATEDIFF(CURDATE(), pm.plan_date) ELSE 0 END) AS max_delay_days,
  -- BOM 维度
  COUNT(bom.id)                         AS bom_total,
  COUNT(CASE WHEN bom.arrival_status = 'ARRIVED' THEN 1 END) AS bom_arrived
FROM `budget_zone` bz
LEFT JOIN `budget_item` bi
  ON bi.zone_id = bz.id AND bi.is_deleted = 0
LEFT JOIN `project_plan` pp
  ON pp.zone_id = bz.id AND pp.is_deleted = 0 AND pp.is_milestone = 0
LEFT JOIN `plan_milestone` pm
  ON pm.task_id = pp.id
LEFT JOIN `bom_items` bom
  ON bom.zone_id = bz.id AND bom.is_deleted = 0
WHERE bz.is_deleted = 0
GROUP BY bz.id, bz.zone_code, bz.zone_name, bz.zone_type, bz.project_id;
```

**性能注意事项**:
- 视图涉及 4 表 JOIN + 多维度聚合，在大数据量下直接查询耗时可能 > 2s
- 生产环境必须通过 Redis 缓存层访问（`cockpit:aggregate:{projectId}`，TTL=5min）
- 若数据量进一步增长，可考虑将视图替换为物化表（定时刷新）

```sql
CREATE TABLE `cockpit_evm_snapshot` (
  `id`          BIGINT         NOT NULL AUTO_INCREMENT,
  `project_id`  BIGINT         NOT NULL COMMENT '项目 ID',
  `snapshot_date` DATE         NOT NULL COMMENT '快照日期（周粒度为每周日，月粒度为每月最后一天）',
  `granularity` VARCHAR(10)    NOT NULL DEFAULT 'WEEK' COMMENT '快照粒度：WEEK / MONTH',
  `pv`          DECIMAL(18,2)  NOT NULL COMMENT '计划价值（元）',
  `ev`          DECIMAL(18,2)  NOT NULL COMMENT '挣值（元）',
  `ac`          DECIMAL(18,2)  NOT NULL COMMENT '实际成本（元）',
  `spi`         DECIMAL(6,4)   NOT NULL COMMENT '进度绩效指数',
  `cpi`         DECIMAL(6,4)   NOT NULL COMMENT '成本绩效指数',
  `eac`         DECIMAL(18,2)  DEFAULT NULL COMMENT '完工估算（AC + (BAC-EV)/CPI）',
  `create_time` DATETIME       NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '快照写入时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_project_date_gran` (`project_id`, `snapshot_date`, `granularity`),
  INDEX `idx_project_date` (`project_id`, `snapshot_date`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='EVM 历史快照表（INSERT-ONLY，定时任务每周写入）';
```

**快照写入时机**:
- 每周日 23:59 由 `@Scheduled` 任务触发，写入本周 EVM 数据（`granularity=WEEK`）
- 每月最后一天 23:59 写入月度快照（`granularity=MONTH`）
- 幂等：UNIQUE KEY 保证同一项目同一日期同一粒度只有一条，重复写入触发 `ON DUPLICATE KEY UPDATE`

## 7. PageGuide 内容

- **这是什么**: 万德项目全景控制表，以「一张表」集中展示区域×预算×计划×BOM×健康度五维数据，配合 EVM 挣值管理指标，让吴耀在 5 分钟内完成所有在建项目的整体健康度评估。

- **解决什么问题**: 替代「查预算系统 → 切计划甘特图 → 看 BOM 清单」的三步切换模式，每日会前准备时间从 30 分钟缩短到 5 分钟。核心痛点：数据分散在 3 个模块，无法横向对比多个项目，EVM 趋势恶化无自动预警机制。

- **快速上手**（3 步）:
  1. **选项目**：顶部项目选择器选择想查看的项目（可多选 2-5 个进行横向对比），点击「刷新」获取最新数据
  2. **看全局**：扫描 EVM 仪表盘 5 张 KPI 卡（绿色=健康 / 橙色=需关注 / 红色=危险），下方矩阵找出红色高亮的异常区域
  3. **下钻问题**：点击矩阵中异常区域的「下钻」→ 查看该区域详细的预算明细 / 里程碑列表 / BOM 到货状态 → 点击「处理问题」跳转到对应模块直接操作

- **权限说明**: 超管（吴耀）可查看所有项目；总监看本负责项目；项目经理只看本项目；多项目对比功能仅超管和总监可用。

## 附录 B：技术决策记录（ADR）

### ADR-1：EVM 计算口径

- **背景**: 挣值管理（EVM）中 PV、EV、AC 的计算口径在业界有多种变体，万德的项目特点（无固定人天工时记录、以里程碑为主要进度凭证）要求制定适合自身业务的口径，并保证各接口使用相同口径。
- **决策（三个指标的唯一口径）**:
  - **PV（计划价值）**: 基于 `plan_milestone` 表中 `plan_date ≤ asOfDate` 的里程碑，按「里程碑权重（均分或按计划价值字段）× 项目总预算 `budget_item.total_amount` 求和」。权重分配方式：当前采用均分（所有里程碑等权重），未来可引入 `plan_milestone.plan_value` 字段按价值加权。
  - **EV（挣值）**: 基于 `plan_milestone` 表中 `actual_date IS NOT NULL`（已实际完成）的里程碑，同样按权重×总预算计算。
  - **AC（实际成本）**: 来自 `budget_item.consumed_amount`，按项目所有 `budget_item` 的 `consumed_amount` 汇总。注意：AC 包含所有已审批完成的采购和报销，不包含未审批的申请。
  - **SPI = EV / PV**，**CPI = EV / AC**，精度 4 位 BigDecimal，前端展示 2 位。
- **执行要求**: `EVM_PV`、`EVM_EV`、`EVM_AC` 仅在 `EvmCalculatorService` 中计算，所有调用方通过该 Service 获取，不允许各接口分别实现计算逻辑。

### ADR-2：5min Redis 缓存 + 数据更新事件主动失效

- **背景**: 全景控制表需要聚合 4 个业务模块的数据（预算/计划/BOM/供应商），JOIN 计算耗时可达 2-5s，无法支持高频刷新。同时，管理层对数据实时性的要求是「5 分钟内感知变化」，而非秒级。
- **决策**:
  - 聚合接口（`/cockpit/aggregate` / `/cockpit/evm` / `/cockpit/health-radar`）统一走 Redis 缓存，TTL=5min。
  - 主动失效（Cache-Aside 模式）：当预算/计划/BOM/供应商排程任一数据发生写入操作时，对应模块 Service 发布 `CockpitCacheEvictEvent`，异步监听器立即删除对应 Redis key。
  - 失效后下一次访问触发 DB 查询重建缓存，响应时间可能较慢（2-5s），前端以 Loading 状态展示。
  - **不采用的方案**：实时流式计算（太复杂，当前数据量不值得引入 Flink/Kafka）；每次访问直接查 DB（响应太慢，管理层体验差）。
- **监控**: 缓存命中率写入监控大盘（Prometheus 指标 `cockpit_cache_hit_total` / `cockpit_cache_miss_total`），若命中率 < 60%，说明数据变更过于频繁，需调整 TTL 或缓存策略。
