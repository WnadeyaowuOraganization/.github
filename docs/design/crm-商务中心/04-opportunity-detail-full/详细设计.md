## 1. 页面概述

- **路由**: `/wande/crm/opportunity/:id`
- **菜单层级**: CRM > 商务中心 > 商机详情
- **权限标识**: `crm:opportunity:detail`
- **所属模块**: `module:fullstack`
- **布局模式**: 左摘要固定区（240px）+ 右侧多 Tab（10个Tab，iframe 切换）
- **核心业务定位**: 商务专员日常最高频操作页，覆盖商机全生命周期（发现→跟进→设计→报价→合同→回款→投标→结案）

### 左侧摘要固定区字段

| 字段 | 说明 |
|------|------|
| 商机名称 | 页面标题，只读 |
| 阶段进度条 | N/C/B/S/P/A/D 七阶段，可视化进度条 |
| 客户名 | 只读，继承自客户表 |
| 合同金额 | 显示最新确认报价金额 |
| 预计签约日 | 可编辑 |
| 负责人 | 只读，继承 |
| 来源 | 矿场发现 / 主动开发 / 转介绍 / 网络询盘 |
| 创建时间 | 只读 |

### 摘要区操作按钮

- **[推进阶段]**: 弹出阶段选择器，只能前进不能后退（除超管）
- **[编辑基本信息]**: 进入 Tab 01 编辑态
- **[放弃跟进]**: 触发 Disqualification 弹窗（§1.1）

### 1.1 放弃跟进弹窗（Disqualification）

- 放弃原因（必填，下拉）: 预算不足 / 竞争对手中标 / 客户需求变化 / 关系资源不足 / 其他
- 详细说明（选填，textarea）
- 确认复选框: "我确认已知晓放弃后商机将关闭"
- 操作结果: 商机阶段变为 `closed_lost`，管道列表中默认隐藏（可通过筛选查看）

## 2. 页面布局

> 每个 §2.x 对应一个 Tab，严格对应原型 10 个 Tab，子 Issue 按 §2.x 拆分。

### 2.1 Tab 01 — 基本信息

- **组件**: `a-form`（双列布局）、`a-tag`（auto 标签标注继承字段）、`a-alert`（AI 预填潜在风险黄色区块）、`a-input-number`（成单预测百分比调整）
- **数据源**:
  - `GET /api/crm/opportunity/{id}` — 加载完整商机字段
  - `PUT /api/crm/opportunity/{id}` — 保存编辑后字段
  - `POST /api/crm/opportunity/{id}/advance-stage` — 推进阶段
  - `POST /api/crm/opportunity/{id}/disqualify` — 放弃跟进
- **交互**:
  - 默认只读，点击[编辑基本信息]进入编辑态，`a-form` 切换为可编辑
  - 自动继承字段（客户名/负责人/所属部门/客户性质）显示蓝色 `auto` tag，不可手动修改
  - AI 预填字段（潜在风险/项目说明/安装条件）黄色区块，标注"AI草稿可修改"
  - 成单预测：系统预测值 + 商务可手动上调/下调
  - 矿场来源区块（仅 `source=mine` 时显示）: project_mine_id / 项目名 / 省市 / 预算区间 / 开放时间 / 矿场评分 / 中标概率
- **代码现状**: 🟡 有基础需改造（`OpportunityController.java` 已存在基础 CRUD，但 AI 预填字段/auto-tag/矿场区块为新增逻辑，前端 Vue 需改造）
- **已有文件**: `backend/.../OpportunityController.java`；`frontend/src/views/wande/crm/opportunity/`

**三级边界规则**:
```
✅ Always:
- 自动继承字段（客户名/负责人/部门）仅从 customer 表同步，前端不允许手动覆盖
- 保存时 crm_opportunity 同步写 crm_opp_change_log（由后端 AOP 拦截，不需前端主动调用）
- 成单预测取值范围 0-100%，存为整数

⚠️ Ask First:
- AI 预填字段（潜在风险/项目说明/安装条件）的 Prompt 模板调整需产品确认
- 矿场评分/中标概率字段来自 project_mine 表，跨模块联动逻辑需确认同步频率

🚫 Never:
- 不允许直接修改商机阶段绕过 advance-stage API（必须走流程校验）
- 不允许删除 crm_opportunity 记录（只能 disqualify/close）
```

### 2.2 Tab 02 — 跟进记录

- **组件**: `a-timeline`（时间线展示历史记录）、`a-modal`（新增跟进弹窗）、`a-tag`（记录类型标签：拜访/电话/邮件/会议）、`a-button`（新增跟进触发）
- **数据源**:
  - `GET /api/crm/opportunity/{id}/followups?page=&size=` — 跟进记录列表（含自动+手动）
  - `POST /api/crm/opportunity/{id}/followups` — 新增跟进记录
  - `DELETE /api/crm/opportunity/{id}/followup/{fid}` — 删除跟进记录（仅本人）
- **交互**:
  - 时间线按时间倒序展示，自动生成记录（流程操作触发，标注"自动"灰色 tag）与手动记录（标注活动类型彩色 tag）混合展示
  - 点击[新增跟进]弹出 Modal：活动类型选择（拜访/电话/邮件/会议）/ 跟进时间 / 内容（textarea）/ 下一步行动 / GPS 定位（拜访时自动捕获）
  - 新增保存后立即插入时间线顶部（乐观更新）
  - 自动记录由后端业务流程写入，前端只读不可删
  - "查看更多"按钮分页加载（每页 10 条）
- **代码现状**: 🔴 需新建（跟进记录独立 Tab，后端需新建 `crm_followup` 表及接口，前端新建 Vue 组件）
- **已有文件**: 暂无专属跟进文件；`crm_activity_log` 由 #4004 维护（本 Tab 写 crm_followup 同时同步 activity_log）

**三级边界规则**:
```
✅ Always:
- 新增跟进记录同步写入 crm_activity_log（log_type='follow'），双写由后端 Service 层负责
- 自动生成记录（系统写入）只读，opertor_type='system'，前端不渲染删除按钮
- 时间线按 follow_time DESC 排序

⚠️ Ask First:
- GPS 定位是否强制要求（拜访类型时）需与商务确认用户接受度
- 跟进内容字数限制（建议 2000 字）是否足够

🚫 Never:
- 不允许修改已保存的跟进记录（只可删除重填）
- 不允许删除自动生成的系统记录
```

### 2.3 Tab 03 — 设计单

- **组件**: `a-table`（设计单列表）、`a-modal`（下设计单表单）、`a-tag`（设计单状态：待接单/设计中/已完成）、`a-button`（新增设计单）
- **数据源**:
  - `GET /api/crm/opportunity/{id}/design-orders` — 获取关联设计单列表（跨模块查询 D3）
  - `POST /api/crm/opportunity/{id}/design-orders` — 发起新设计单（触发 D3 模块）
- **交互**:
  - 列表列：设计单号 / 种类（X-规划类等）/ 设计师 / 状态 / 创建日期 / 交付附件
  - 点击[下设计单]弹出简化表单（设计需求描述 / 参考附件上传 / 预计交付日），客户/项目等级/预算自动从商机继承
  - 点击设计单号跳转 D3 模块详情页（新 Tab 打开）
  - 下设计单后商机阶段自动推进到"商务接触"（若还未到）
- **代码现状**: 🔴 需新建（D3 模块跨模块联动接口需新建；CRM 侧 `crm_design_order` 关联表需新建）
- **已有文件**: D3 模块 Controller 在独立业务域，需通过内部服务调用或共享表关联

**三级边界规则**:
```
✅ Always:
- 设计单列表数据从 D3 模块同步，不在 CRM 数据库独立存储完整设计数据
- 下设计单时自动带入商机 ID 作为 ref_opportunity_id
- 设计单状态变更时通过事件回调更新商机内的"是否有设计单"字段

⚠️ Ask First:
- D3 设计单接口调用方式（直接 HTTP 内部调用 vs 共享数据库视图）需架构确认
- 一个商机是否允许多个设计单同时进行中

🚫 Never:
- 不允许在 CRM 内编辑/删除 D3 设计单（只可发起，后续操作跳转 D3 模块）
```

### 2.4 Tab 04 — 报价

- **组件**: `a-table`（报价单列表）、`a-modal`（新建报价单表单）、`a-tag`（状态：草稿/审批中/已通过/作废）、`a-steps`（审批进度）
- **数据源**:
  - `GET /api/crm/opportunity/{id}/quotes` — 报价单列表
  - `POST /api/crm/opportunity/{id}/quotes` — 新建报价单
  - `POST /api/crm/opportunity/{id}/quote/{qid}/submit-approval` — 提交审批
  - `PUT /api/crm/opportunity/{id}/quote/{qid}/void` — 作废报价单
- **交互**:
  - 列表列：报价编号 / 报价金额 / 状态 / 审批人 / 日期 / 操作
  - 点击[新建报价]弹出表单：报价金额 / 有效期 / 备注
  - 保存为草稿后可点击[提交审批]触发审批流（审批人为董事长/吴耀）
  - 状态"审批中"时显示审批步骤组件（a-steps）
  - 报价审批通过后，合同 Tab 的[新建合同]按钮解锁
- **代码现状**: 🔴 需新建（报价单数据模型及审批流联动均需新建）

**三级边界规则**:
```
✅ Always:
- 同一商机同时只能有一个"审批中"状态的报价单
- 报价单审批通过后不可再修改金额（只能作废重新提交）
- 提交审批同步触发审批体系（#3161 企微审批贯通）

⚠️ Ask First:
- 报价金额与商机预算金额差异超过 20% 是否需要额外审批节点
- 审批驳回后是否允许修改后重新提交同一单号

🚫 Never:
- 不允许绕过审批直接将报价状态改为"已通过"
- 不允许删除已提交审批的报价单（只可作废）
```

### 2.5 Tab 05 — 合同

- **组件**: `a-table`（合同列表）、`a-modal`（新建合同表单）、`a-tag`（状态：草稿/签约中/已签/已终止）、`a-upload`（合同扫描件上传）
- **数据源**:
  - `GET /api/crm/opportunity/{id}/contracts` — 合同列表
  - `POST /api/crm/opportunity/{id}/contracts` — 新建合同
  - `PUT /api/crm/opportunity/{id}/contract/{cid}` — 更新合同状态/上传附件
- **交互**:
  - 列表列：合同编号 / 合同金额 / 签约日期 / 盖章状态 / 状态 / 操作
  - [新建合同]仅在有已通过报价单时可点击（否则 disabled + tooltip 提示"需先完成报价审批"）
  - Sprint-1 只做基础创建：合同编号（自动生成）/ 合同金额（预填自报价）/ 签约日期 / 备注 / 合同扫描件上传
  - 盖章状态：未盖章 / 已甲方盖章 / 已双方盖章
- **代码现状**: 🔴 需新建（Sprint-1 只做基础字段，高级合同管理功能后续迭代）

**三级边界规则**:
```
✅ Always:
- 创建合同前校验该商机是否存在已通过的报价单
- 合同编号格式：HT-YYYY-XXXX（后端自动生成，前端只读）
- 新建合同后商机阶段自动推进到"签约跟进"（若当前阶段在此之前）

⚠️ Ask First:
- 合同扫描件是否上传到 S3（wande-nas-sync 桶）还是本地存储
- 合同终止时是否需要审批流

🚫 Never:
- 不允许删除已签合同（只可标记"已终止"）
- Sprint-1 不实现合同模板/电子签章，后续再迭代
```

### 2.6 Tab 06 — 回款

- **组件**: `a-table`（回款节点计划列表）、`a-table`（收款记录列表）、`a-modal`（新增收款记录）、`a-tag`（状态：未到期/待收/已收/逾期）、`a-alert`（逾期红色警告）
- **数据源**:
  - `GET /api/crm/opportunity/{id}/payment-plans` — 回款节点计划列表
  - `GET /api/crm/opportunity/{id}/payment-records` — 收款记录列表
  - `POST /api/crm/opportunity/{id}/payment-records` — 新增收款记录
  - `PUT /api/crm/opportunity/{id}/payment-plan/{pid}/status` — 更新回款节点状态
- **交互**:
  - 签约后自动生成回款计划（完结款/发货款/验收款/审计款/质保金五节点），未签约时显示空状态
  - 回款节点按截止日期排序，逾期节点标红（截止日 < 今日且未收）
  - [新增收款记录]弹窗：关联节点 / 收款金额 / 到账日期 / 银行流水号 / 备注
  - 页面右上角[填周报]入口：填写本合同本周回款情况（联动回款主页 §8 回款跟踪）
- **代码现状**: 🔴 需新建（回款节点模型及逾期预警逻辑均需新建）

**三级边界规则**:
```
✅ Always:
- 回款计划在合同状态变为"已签"时由后端自动生成（不依赖前端触发）
- 逾期判断：截止日 < 系统当日 AND status != 'received'，后端返回 is_overdue 字段
- 收款记录金额累计不得超过合同总金额（后端校验）

⚠️ Ask First:
- 五节点回款计划的比例是否固定（如 30/30/20/10/10）或由商务自定义
- 周报填写是否与记录中心的周报体系复用同一接口

🚫 Never:
- 不允许在未签约状态下手动创建回款计划
- 不允许将收款金额超过对应节点应收金额
```

  - 顶部分类筛选（全部 / 招标文件 / 设计图纸 / 合同 / 其他）

  - 点击文件名触发下载（通过 presigned URL 直接下载，不走后端中转）

- **代码现状**: 🔴 需新建（附件独立存储于 `crm_opp_attachment` 表，S3 上传集成）

**三级边界规则**:
```
✅ Always:

- 下载链接有效期：1 小时（presigned URL）

⚠️ Ask First:
- 是否需要附件预览功能（PDF 内嵌预览）

🚫 Never:
- 不允许上传可执行文件（.exe/.bat/.sh）
- 不允许覆盖同名文件（自动加时间戳后缀）
```

### 2.9 Tab 09 — 流程监控

- **组件**: `a-table`（审批流列表）、`a-tag`（状态：待提交/审批中/已通过/已驳回）、`a-steps`（审批步骤展开）、`a-button`（催办按钮）
- **数据源**:
  - `GET /api/crm/opportunity/{id}/approval-flows` — 该商机下所有审批流列表
  - `POST /api/crm/approval/{flow_id}/remind` — 催办（向当前处理人发通知）
- **交互**:
  - 列表列：流程类型（报价/合同/投标）/ 关联单号 / 当前节点 / 处理人 / 发起时间 / 等待天数 / 状态 / 操作
  - 等待天数 ≥ 5 天：行标红 + 等待天数数字标红
  - [催办]按钮：点击后调用 remind API，向处理人发企微通知，同一流程同一天只能催办一次（后端限频）
  - [详情]展开该审批流完整步骤（`a-steps`）
  - 商机无任何审批流时显示空状态
- **代码现状**: 🟡 有基础需改造（审批体系 #3161-#3166 已规划，但商机维度聚合视图需新建）

**三级边界规则**:
```
✅ Always:
- 审批流数据从审批体系模块读取，不在 CRM 独立存储
- 催办限频：同一 flow_id 同一自然日最多触发 1 次（后端 Redis 限流）
- 等待天数按工作日计算（排除法定节假日）

⚠️ Ask First:
- 催办通知是否同时发企微 + 站内通知，还是只选其一
- 是否需要展示已通过/已驳回的历史审批流

🚫 Never:
- 不允许非审批流发起人进行催办（校验 operator_id == flow.initiator_id）
```

### 2.10 Tab 10 — 变更日志

- **组件**: `a-table`（变更日志列表）、`a-select`（字段类型筛选）、`a-tag`（变更类型：字段变更/阶段推进）
- **数据源**:
  - `GET /api/crm/opportunity/{id}/change-logs?field=&page=&size=` — 变更日志列表（支持按字段筛选）
- **交互**:
  - 列表列：变更时间 / 变更字段 / 变更前值 / 变更后值 / 操作人
  - 阶段推进记录单独标注（tag 加粗蓝色，如"商务接触 → 招标准备"）
  - 系统自动操作记录（矿场分配/自动推进）操作人显示为"系统"
  - 顶部字段筛选下拉（全部 / 商机阶段 / 成单预测 / 合同金额 / 负责人 / 其他字段）
  - 按 change_time DESC 排序，分页每页 20 条
- **代码现状**: 🔴 需新建（`crm_opp_change_log` 表需新建，后端 AOP 拦截字段变更自动写入）

**三级边界规则**:
```

- 所有 crm_opportunity 字段变更均捕获，包括系统触发的自动变更
- 变更日志只读，不可修改或删除

⚠️ Ask First:
- 变更日志保留期限（建议永久保留或 3 年）
- 是否需要导出变更日志为 Excel

🚫 Never:
- 不允许手动创建变更日志记录（必须由 AOP 自动生成）
- 不允许前端直接调用 insert change_log 接口
```

## 3. API 契约

### 3.1 Tab 01 — 基本信息 API

#### GET `/api/crm/opportunity/{id}`
- **参数**: `id` (path, bigint) — 商机 ID
- **返回**: 完整商机对象（含矿场来源区块、AI 预填字段、自动继承标注）

#### PUT `/api/crm/opportunity/{id}`
- **参数**: body `{ budget_amount, sign_date_plan, win_rate_manual, risk_note, project_desc, install_condition, ... }`
- **返回**: 更新后商机对象；副作用：写 crm_opp_change_log

#### POST `/api/crm/opportunity/{id}/advance-stage`
- **参数**: body `{ target_stage: 'C'|'B'|'S'|'P'|'A'|'D' }`
- **返回**: 更新后商机阶段；校验：只能前进阶段，不可后退

#### POST `/api/crm/opportunity/{id}/disqualify`
- **参数**: body `{ reason: string, detail?: string }`
- **返回**: 商机状态变为 `closed_lost`
