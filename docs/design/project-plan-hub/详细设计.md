- **菜单层级**: 项目中心 > 项目计划
- **权限标识**: `wande:plan:*`
- **所属模块**: `module:fullstack`
- **核心表**: `project_plan` / `plan_milestone` / `plan_template` / `plan_supplier_schedule` / `plan_dependency` / `plan_change_log` / `plan_alert_rule`
- **原型架构**: 5 Tab 多文件架构（`index.html` + 5 个 `tab-*.html`）
- **依赖服务**:
  - BOM 模块（`ProjectBomController`，变更影响分析事件源）

  - networkx Python CPM 计算服务（关键路径，Pipeline 微服务调用）
  - 企微群机器人（预警推送，频控 24h 不重复）

### 1.1 业务背景

万德项目计划长期依赖 Excel Gantt 图和现场口头协调，存在三大核心痛点：

1. **供应商排程与主计划脱节**：供应商自行维护排程，与甲方主计划无接口点，延期发现滞后 1-2 周。
2. **BOM 变更影响主计划的连锁反应不可见**：BOM 新增物料或交期变更后，哪些计划任务会受影响、延期多少天，目前靠项目经理人工评估。
3. **告警升级机制缺失**：延期超过 3 天的任务只有项目经理知道，总监/吴耀无法自动感知，导致决策介入太晚。

项目计划管理模块的目标：
1. **甘特图主计划**：以可视化甘特图管理主计划，关键路径 CPM 自动计算，里程碑高亮。
2. **供应商排程集成**：供应商在专属视图录入自己的交货排程，系统自动比对与主计划的偏差。
3. **BOM 联动影响分析**：BOM 变更时自动推算对哪些计划任务产生影响，一键应用变更。
4. **多级预警升级**：延期 >3 天通知负责人，>7 天通知总监，>14 天通知吴耀；企微推送频控避免轰炸。

### 1.2 用户角色

| 角色 | 典型操作 | 权限范围 |
|------|---------|---------|
| 吴耀（超管） | 查看管理层仪表盘 / 设置告警规则 | 全量只读 + 告警配置 |
| 项目经理 | 编制主计划 / 审批供应商排程 / 触发 BOM 影响分析 | 本项目全操作 |
| 供应商联络员 | 录入供应商交货排程 / 标记接口点 | 本项目供应商排程写权限 |
| 管理层（总监） | 查看多项目仪表盘 / 接收升级告警 | 只读 + 接收通知 |

### 2.1 主计划（Tab1 — tab-master-plan.html）— 对应 Issue #3195 #3197 #3198

**功能描述**: 以甘特图展示项目主计划，支持基于模板创建计划、任务拖拽调整时间、里程碑标记、关键路径高亮。项目经理在此做计划编制、调整和版本管理。

- **组件**:
  - `PlanGanttChart`（甘特图主体：左侧任务列表树 + 右侧时间轴横道图，支持展开/折叠任务分组）
  - `MilestoneMarker`（里程碑钻石图标，置于甘特条上方，hover 显示里程碑名称/预计日期/实际日期）
  - `CriticalPathHighlight`（关键路径高亮：红色甘特条 + 浮层展示 CPM 总浮时为 0 的任务链）
  - `PlanTemplateSelector`（计划模板选择器：内置项目类型模板（主题公园/水上乐园/儿童乐园）+ 自定义模板）
  - `TaskEditDrawer`（任务详情抽屉：任务名 / 负责人 / 计划开始/结束日期 / 实际开始/结束日期 / 进度% / 前置任务 / 关联 BOM 条目）
  - `PlanVersionPanel`（计划版本管理：快照列表 / 对比视图 / 回退功能）
  - `PlanToolbar`（工具栏：新建任务 / 按模板生成 / 导出 PDF / 导出 MS Project 格式 / 计算关键路径）

- **数据源**:
  - `POST /wande/plan/master`（按模板生成主计划，`projectId` + `templateId`，返回完整任务树）
  - `GET /wande/plan/master`（获取主计划任务树，`projectId` + `version`）
  - `PUT /wande/plan/master/task`（更新单任务：日期 / 进度 / 负责人 / 前置任务）
  - `GET /wande/plan/master/critical-path`（获取关键路径计算结果，CPM）

- **交互**:
  - 进入 Tab1 时自动加载当前项目最新版本计划，切换版本按钮在工具栏右侧

  - 里程碑图标点击 → 内联编辑里程碑名称和预计日期
  - 点击「计算关键路径」→ 调用 CPM 接口（内部调用 networkx Python 服务）→ 关键链任务变红色甘特条
  - 任务行右键菜单：新增子任务 / 编辑 / 删除 / 拆分任务 / 设为里程碑
  - 「按模板生成」→ 弹窗选择模板 → 预览模板任务结构 → 确认后基于模板创建初始计划（已有计划时提示是否覆盖）

### 2.2 供应商排程（Tab2 — tab-supplier.html）— 对应 Issue #3194 #3196 #3201

**功能描述**: 供应商联络员在此录入各自的物料交货排程（按 SKU/批次），系统自动与主计划的「物料需求节点」比对，生成排程偏差矩阵。项目经理可在此标记「接口点」（主计划与供应商排程的对接时间节点）。

- **组件**:
  - `SupplierScheduleTable`（供应商排程表格：供应商 / 物料 SKU / 批次 / 计划交货日 / 实际确认日 / 偏差天数 / 主计划需求节点 / 状态）
  - `ScheduleCalendar`（排程日历视图：以日历形式展示各供应商交货节点，颜色区分供应商）
  - `DependencyMatrix`（依赖矩阵热力图：行=主计划任务，列=供应商排程批次，单元格=是否存在依赖关系及偏差状态）
  - `InterfacePointModal`（接口点配置弹窗：选择主计划任务 + 选择供应商排程批次 + 设置缓冲天数 + 负责人）
  - `ScheduleImportBtn`（批量导入供应商排程 Excel：字段映射 + 冲突检测）
  - `DeviationAlertPanel`（偏差告警面板：列出偏差 > 3 天的条目，支持逐条确认或推送告警）

- **数据源**:
  - `GET /wande/plan/supplier-schedule`（供应商排程列表，`projectId` + `supplierId` + 状态）
  - `POST /wande/plan/supplier-schedule`（新建/批量导入排程，含字段映射解析）
  - `GET /wande/plan/supplier-schedule/dependency-matrix`（依赖矩阵数据）
  - `POST /wande/plan/supplier-schedule/interface-point`（创建接口点）

- **交互**:
  - 排程表格按供应商分组展示，默认折叠，点击展开看具体 SKU/批次
  - 偏差天数列：正数（延迟）红色标注，负数（提前）绿色标注，偏差 > 3 天行背景高亮橙色
  - 切换「日历视图」→ 显示当月交货节点日历，不同供应商用不同颜色点标注
  - 依赖矩阵：行列交叉点有「接口点」标记的显示绿色 ✓，无关联显示灰色，有偏差风险的显示橙色 △
  - 「标记接口点」→ 弹窗选择主计划任务 + 供应商批次 → 设置缓冲天数（缓冲期内偏差不触发告警）
  - 供应商联络员录入排程 → 行末出现「确认」按钮（仅项目经理可见）→ 项目经理确认后状态锁定，偏差触发告警

- **代码现状**: 🟡 有基础需改造
  - `frontend/src/views/project/plan/SupplierCalendar.vue` 已有日历视图基础，需扩展表格视图和依赖矩阵

### 2.3 BOM 联动（Tab3 — tab-bom-link.html）— 对应 Issue #3203

**功能描述**: 当 BOM 发生物料变更（新增/删减/交期变更）时，系统分析该变更对主计划任务的影响范围（哪些任务依赖该物料、预计延期多少天），以可视化方式展示影响链，项目经理确认后一键应用变更（自动调整相关计划任务日期）。

- **组件**:
  - `BomChangeEventList`（BOM 变更事件列表：变更类型 / 物料 SKU / 变更内容 / 影响评估状态 / 操作时间）
  - `ImpactAnalysisPanel`（影响分析面板：树形展示受影响任务链，每节点显示预计延期天数和累计风险）
  - `ImpactApplyModal`（应用变更确认弹窗：逐条列出将要调整的任务 + 调整量 + 是否连锁调整后续任务）
  - `BomLinkMatrix`（物料-任务关联矩阵：行=主计划任务，列=关联 BOM 物料，单元格显示依赖关系）
  - `ChangeHistoryLog`（变更历史日志：过去所有 BOM 变更事件及其影响应用记录）

- **数据源**:
  - `POST /wande/plan/bom-link/impact-analysis`（触发影响分析，`bomChangeEventId`，异步分析后返回影响树）
  - `POST /wande/plan/bom-link/apply-change`（应用变更，一键调整受影响任务日期）

- **交互**:
  - BOM 变更事件由 `ProjectBomController` 的变更操作发布 `BomChangedEvent`，计划模块监听后自动在列表中追加新事件（WebSocket 推送或前端轮询）
  - 点击某事件「分析影响」→ Loading（异步分析，最长 10s）→ 右侧展开影响树，按「延期天数」由高到低排序
  - 影响树节点点击 → 跳转到甘特图（Tab1）对应任务高亮（Hash 定位）
  - 「应用变更」→ 弹窗逐条展示将调整的任务 → 用户可取消选中不需要联动的任务 → 确认 → 批量 PUT 任务日期 → plan_change_log 记录
  - 「忽略」→ 该变更事件标记为「已忽略」，不再提示，但在变更历史中保留记录
  - 关联矩阵展示当前主计划所有任务与 BOM 物料的依赖关系，用于首次建立关联

- **代码现状**: 🟡 有基础需改造（`ProjectBomController.java` 为 BOM 事件源，监听逻辑需新建）
  - `backend/.../execution/ProjectBomController.java` 已有 BOM 变更接口，需在变更方法中发布 `BomChangedEvent`
  - `PlanBomLinkService`（新建）：订阅 `BomChangedEvent`，执行影响分析（CPM 子集计算），写入 `plan_change_log`
  - 影响分析调用 networkx Python 服务（与关键路径共用同一 Pipeline 服务接口）

- **已有文件**:
  - `backend/.../execution/ProjectBomController.java`（BOM 事件源，需增加 `@ApplicationEvent` 发布）

- **三级边界**:

```
✅ Always（必须做）:
- BOM 影响分析结果必须包含「延期天数」和「影响任务路径」（任务链），不允许仅返回受影响任务列表
- 应用变更后，所有被调整的任务必须写入 plan_change_log（action=BOM_IMPACT_APPLY，含 before/after 日期）
- 影响分析异步执行，若 10s 内未完成则返回超时，提示用户稍后刷新（不阻塞前端）

⚠️ Ask First（需确认再做）:
- 影响分析自动应用（无需人工确认直接调整计划）——自动化风险高，当前保持人工确认模式
- 支持批量应用多个 BOM 变更事件的影响（当前一次只能应用一个事件）

🚫 Never（禁止做）:
- 不得在不经用户确认的情况下自动修改主计划任务日期（即使影响分析完成也需要用户点击「应用变更」）
- 不得忽略有「接口点」依赖的任务（接口点关联的任务在影响分析中必须包含在影响树中）
```

### 2.4 预警引擎（Tab4 — tab-alert.html）— 对应 Issue #3199 #3200

**功能描述**: 系统根据配置的告警规则，实时监测主计划任务进度，对延期/即将延期的任务分级告警，通过企微推送到对应级别的负责人。支持告警规则自定义（延期阈值、升级路径、推送频率）。

- **组件**:
  - `AlertRuleList`（告警规则列表：规则名 / 触发条件 / 升级路径 / 推送频率 / 状态 / 操作）
  - `AlertRuleEditModal`（告警规则编辑弹窗：条件配置（延期天数/里程碑紧迫性）+ 升级阶梯设置 + 推送群配置）
  - `ActiveAlertTable`（当前活跃告警列表：任务 / 延期天数 / 当前告警级别 / 最近推送时间 / 负责人 / 状态）
  - `AlertTimeline`（单任务告警时间线：该任务从初次告警到升级的完整事件流）
  - `AlertStatsDashboard`（告警统计面板：今日新增 / 已处理 / 上升趋势 / 平均解决时间 4 张卡片）
  - `WechatPushConfigPanel`（企微推送配置：群 webhook 地址 + 消息模板 + 频控设置）

- **数据源**:
  - `GET /wande/plan/alert`（当前活跃告警列表，`projectId` + `alertLevel` + `status`）
  - `POST /wande/plan/alert/wechat-config`（配置企微推送参数）
  - `POST /wande/plan/alert/escalate`（手动触发告警升级，测试用途，超管专用）

- **交互**:
  - 告警规则列表展示系统内置规则（不可删除）+ 项目经理自定义规则（可编辑/禁用）
  - 点击「新建规则」→ 弹窗填写：条件（延期 N 天）+ 阈值梯度（3天→负责人 / 7天→总监 / 14天→超管）+ 推送方式（企微/内部消息/双发）
  - 当前告警列表按延期天数倒序，超过 14 天的行背景红色；7-14 天橙色；3-7 天黄色
  - 点击某告警行「查看告警历史」→ 展开单任务告警时间线（初次告警 → 升级到总监 → 升级到超管的完整记录）
  - 「标记已处理」：项目经理填写处理说明后，该告警状态变为「已处理」，企微推送一条「已解决」通知
  - 企微推送配置：填入群 webhook URL → 「测试推送」验证 → 保存；消息模板中变量：`{projectName}/{taskName}/{delayDays}/{assignee}`

## 7. PageGuide 内容

- **这是什么**: 万德项目计划管理中心，以甘特图管理主计划，联动供应商排程和 BOM 变更，全自动多级预警升级到对应层级负责人。

- **解决什么问题**: 替代 Excel Gantt 和口头协调模式。核心痛点：供应商交货延期 1 周才被发现，BOM 变更后不知道哪些计划任务会受影响，告警只到项目经理层不会自动升级到总监/吴总。

- **快速上手**（3 步）:
  1. **建主计划**：进入「主计划」Tab → 选择项目 → 按模板生成初始甘特图（主题乐园/水上乐园模板）→ 点击「计算关键路径」自动红色高亮关键链
  2. **录供应商排程**：进入「供应商排程」Tab → 录入或导入 Excel 排程 → 标记接口点（设置缓冲天数）→ 依赖矩阵自动显示偏差风险
  3. **看预警**：进入「预警引擎」Tab → 查看活跃告警（已自动按延期天数分级）→ 处理完成后标记「已处理」，告警关闭

- **权限说明**: 项目经理有本项目全操作权限；供应商联络员只能录入排程；总监和吴总通过管理层仪表盘看多项目汇总视图。

## 附录 B：技术决策记录（ADR）

### ADR-1：计划事件总线 — Spring ApplicationEvent

- **背景**: BOM 变更、任务延期、审批通过需要触发多个下游动作（影响分析、告警更新、日志写入），如果在主流程中串行调用所有 Service 会导致耦合度高、响应慢、部分失败难以处理。
- **决策**: 所有触发点发布 Spring ApplicationEvent，下游动作通过 `@EventListener` 订阅：
  - `BomChangedEvent`：`ProjectBomController` 发布 → `PlanBomLinkService` 订阅（影响分析）+ `PlanChangeLogService` 订阅（日志）
  - `PlanTaskDelayedEvent`：`PlanAlertEngine`（定时任务）发布 → `AlertService` 订阅（告警记录）+ `WechatWorkNotifyService` 订阅（推送）
  - `MilestoneCompletedEvent`：`PlanService` 发布 → `DashboardCacheService` 订阅（缓存失效）
- **约束**: 核心业务逻辑（影响分析写 DB）同步监听；通知推送使用 `@Async` 异步监听，不影响主流程响应时间

### ADR-2：关键路径用 networkx Python 库做 CPM 计算

- **背景**: CPM（关键路径法）需要图算法支持，Java 生态无成熟轻量级图计算库，且 CPM 计算是 CPU 密集型，不适合与主业务 Service 混部。
- **决策**: 独立 Python Pipeline 微服务（Flask），使用 `networkx` 库实现 CPM 计算（`DiGraph` + `longest_path` + 浮时计算）。Java `PlanCpmService` 通过 HTTP 调用 Pipeline 服务（超时 5s）。
- **接口格式**: Java 传 `{nodes: [{id, duration}], edges: [{from, to, lag}]}` → Pipeline 返回 `{criticalPath: [...], floats: {...}}`
- **降级策略**: Pipeline 服务不可用时，Java 层返回 503 并在 Redis 存标记，前端提示「关键路径计算不可用，请稍后重试」；不影响甘特图展示和其他操作

### ADR-3：企微推送频控 — 同一任务同一级别 24h 不重复

- **背景**: 预警引擎每 30min 扫描一次，若没有频控，同一延期任务每 30min 就会重复推送告警到企微，造成告警轰炸（一个任务延期 3 天会推送 144 次）。
- **决策**: Redis 频控 key 格式 `plan:alert:{taskId}:{level}:{date}`，TTL=24h。每次推送前检查 key 是否存在，存在则跳过推送（仅更新 DB 告警记录，不推送企微）。
- **升级逻辑**: 任务延期天数超过 L2 阈值（默认 7 天）时，在 L2 的 Redis key 到期后才推送 L2 级别告警（不是实时升级）。即：L2 最快在达到阈值后 24h 发送第一条告警，之后每 24h 一条（直到任务标记为已处理）。
- **特例**: 任务延期超过 L3 阈值（14 天）时，忽略 L2 频控，直接推送 L3（超管级别，确保吴总及时知晓）
