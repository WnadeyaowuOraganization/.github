## 1. 页面概述

- **路由**: `/wande/budget`
- **菜单层级**: 综合管理中心 > 总控预算
- **权限标识**: `wande:budget:*`
- **所属模块**: `module:fullstack`
- **核心表**: `budget_zone` / `budget_subject` / `budget_item` / `budget_template` / `budget_transfer` / `budget_baseline` / `deposit_ledger` / `budget_param`
- **原型架构**: 6 Tab 多文件架构（`index.html` + 6 个 `tab-*.html`）
- **依赖服务**:
  - 项目中心模块（共享 `zone_id`，避免数据孤岛）
  - BOM 模块（`bom_items` 联动余额条）
  - Spring ApplicationEvent 事件总线（调剂审批通过 → 矩阵更新）
  - 企微群机器人（预算超支告警推送）
  - Flyway（`db/migration_wande_ai/`，8 个迁移脚本）

### 1.1 业务背景

万德项目预算长期面临"设计/编制/执行"三阶段数据割裂问题：设计阶段的预算在 Excel 中管理，编制阶段在明道云填报，执行阶段的采购和报销在钉钉审批，三个平台数据无法自动联通。总控预算二维矩阵的目标是：

1. **统一二维管控轴**：以「区域（Zone）× 科目（Subject）」构建矩阵，让吴耀一眼看出哪个区域的哪类费用超支。
2. **三阶段贯穿**：设计阶段设定限额（软提醒）→ 编制阶段锁定数字（硬限额）→ 执行阶段采购/报销硬拦截。
3. **调剂有据可查**：区域间预算调剂必须经过审批流，调剂记录完整保留，矩阵自动更新。
4. **BOM 联动**：采购单绑定预算行后，剩余可用额度实时显示在 BOM 清单中，采购员实时感知余额。

### 1.2 用户角色

| 角色 | 典型操作 | 权限范围 |
|------|---------|---------|
| 吴耀（超管） | 查看全部矩阵 / 审批调剂 / 配置参数 | 全量读写 |
| 项目经理 | 编制本项目预算 / 申请调剂 / 查看执行仪表盘 | 本项目全操作 |
| 采购员 | 创建采购单（余额校验）/ 查看 BOM 联动 | 消费类写权限 + 只读矩阵 |
| 财务专员 | 查看汇总 / 导出报表 / 保证金台账 | 只读 + 导出 |

### 2.1 区域管理（Tab1 — tab-zones.html）— 对应 Issue #3208 #3209 #3217

**功能描述**: 项目经理在此创建和管理预算区域（Zone），支持从招投标方案批量导入区域列表；区域与 BOM 区域、计划区域共用同一个 `zone_id`，实现跨模块数据打通。

- **组件**:
  - `ZoneListTable`（区域列表表格：区域编码 / 区域名称 / 类型 / 归属项目 / 面积 / 预算总额 / 状态 / 操作按钮）
  - `ZoneCreateModal`（新建区域弹窗：编码 / 名称 / 类型（室内/室外/水上/综合）/ 关联项目 / 备注）
  - `ZoneImportModal`（从招投标方案批量导入：上传 Excel + 字段映射预览 + 冲突提示）
  - `ZoneDetailDrawer`（右侧抽屉：区域详情 + 关联 BOM 条目数 + 计划任务数 + 已用预算比例）
  - `ZoneBudgetSummaryCard`（区域预算快览卡：本项目全部区域的已编/已用/余额三栏汇总）

- **数据源**:
  - `GET /wande/budget/zones`（区域列表，支持 `projectId` / `zoneType` / `status` 筛选，分页）
  - `POST /wande/budget/zones`（新建区域）
  - `PUT /wande/budget/zones/{zoneId}`（编辑区域）
  - `DELETE /wande/budget/zones/{zoneId}`（逻辑删除，需检查无关联预算行）
  - `POST /wande/budget/zones/import-bidding`（从招投标 Excel 批量导入区域）

- **交互**:
  - 列表支持按区域名称关键字搜索、按项目 / 区域类型下拉筛选
  - 点击「新建区域」→ 弹窗填写 → 提交后刷新列表
  - 点击「导入招投标方案」→ 上传 Excel → 弹窗预览解析结果（高亮已存在同编码的区域，可选择跳过或覆盖）→ 确认导入
  - 点击某行区域名 → 右侧展开详情抽屉，显示该区域在 BOM、计划、预算三个模块的关联数据摘要
  - 删除区域：若区域已有预算编制行（`budget_item` 记录），则弹出警告「该区域已有 N 条预算行，删除后这些预算行将关联失效，是否继续？」

- **代码现状**: 🟡 有基础需改造
  - 已有 `backend/ruoyi-modules/wande-ai/src/main/java/org/ruoyi/wande/budget/BudgetItemServiceImpl.java`，原逻辑不含 `zone_id` 维度，需在 Service 层新增区域管理方法
  - 已有 `ProjectBudgetController.java`，需扩展 `/budget/zones` 前缀路由
  - Excel 导入逻辑可复用现有 `ExcelImportUtil`（参考 BOM 模块已有实现）

- **已有文件**:
  - `backend/ruoyi-modules/wande-ai/src/main/java/org/ruoyi/wande/budget/BudgetItemServiceImpl.java`
  - `backend/ruoyi-modules/wande-ai/src/main/java/org/ruoyi/wande/budget/ProjectBudgetController.java`

- **三级边界**:

```
✅ Always（必须做）:
- zone_id 必须与 BOM 模块和计划模块共享同一主键，不得各模块自建独立 zone 表
- 区域编码唯一性在 DB 层（项目范围内 UNIQUE KEY）+ Service 层双重校验
- 导入招投标方案时，已存在同编码区域默认「跳过」，需用户主动勾选「覆盖」才允许更新
- 逻辑删除（is_deleted=1），不允许物理删除，历史预算数据必须可追溯

⚠️ Ask First（需确认再做）:
- 跨项目复制区域模板（如将 A 项目区域结构复制到 B 项目），需评估 zone_id 是否需要新生成
- 修改区域类型枚举值（影响下游 BOM 和计划模块的枚举联动）
- 批量归档区域（将已完工区域改为 ARCHIVED 状态）需评估对历史报表的影响

🚫 Never（禁止做）:
- 不得物理删除区域记录（zone_id 是其他模块的关联外键）
- 不得允许非本项目用户修改他人项目的区域（Service 层校验 projectId 归属）
- 不得在 Excel 导入时跳过字段映射校验（区域编码和区域名称为必填，空值直接报错拒绝导入）
```

### 2.2 矩阵编制（Tab2 — tab-matrix.html）— 对应 Issue #3210 #3214 #3218 #2260 #2287

**功能描述**: 以「区域 × 预算科目」为轴，展示二维矩阵表格。项目经理在矩阵单元格中填写预算金额，提交后锁定为编制基准。支持按行（区域）汇总和按列（科目）汇总，单元格支持添加备注。

- **组件**:
  - `BudgetMatrixTable`（核心二维矩阵表格：冻结首列「区域名」+ 冻结首行「科目」+ 可滚动内容区 + 行/列汇总行/列）
  - `MatrixCellEditor`（单元格 inline 编辑：金额输入 + 备注气泡图标）
  - `MatrixToolbar`（工具栏：项目选择 / 年度选择 / 状态标签 / 导出 Excel / 提交编制 / 锁定按钮）
  - `SubjectConfigPanel`（科目配置侧边栏：增删预算科目 / 排序 / 是否必填）
  - `MatrixDiffViewer`（矩阵版本对比视图：当前版本 vs 上一版本，差异单元格高亮）
  - `BudgetSummaryFooter`（底部汇总条：按行/列展示合计金额 + 与设计阶段限额的偏差）

- **数据源**:
  - `GET /wande/budget/matrix`（获取矩阵数据，`projectId` + `year`，返回全部单元格数据）
  - `POST /wande/budget/matrix`（提交整张矩阵（批量 upsert），保存为「草稿」或「已编制」）
  - `PUT /wande/budget/matrix/cell`（单单元格更新，`zoneId` + `subjectId` + `amount` + `remark`）

- **交互**:
  - 矩阵默认展示当前项目当年度的数据，支持切换项目/年度
  - 单元格点击进入编辑模式（金额输入框），失焦后自动保存草稿（PUT /cell 接口）
  - 行/列汇总实时更新（前端计算），不依赖接口
  - 单元格金额超过「设计阶段限额」时，单元格背景变黄（软提醒）
  - 单元格金额超过「设计阶段限额 × 110%」（可配置）时，背景变红，不允许提交编制（硬限额校验在 Service）
  - 点击「提交编制」→ Service 层全量校验（无红色超限单元格）→ 状态变为「已编制」，后续修改需走调剂流程
  - 点击「导出 Excel」→ 生成带汇总行的矩阵 Excel 文件（文件名：`{项目名称}-预算矩阵-{YYYYMMDD}.xlsx`）

- **代码现状**: 🟡 有基础需改造
  - `BudgetItemServiceImpl.java` 已有按项目查询预算行逻辑，但没有二维矩阵结构（zone × subject 轴），需重构返回格式
  - 矩阵单元格 upsert 逻辑需新建，已有的 `ProjectBudgetController` 是行级 CRUD，需新增 cell 级端点

- **已有文件**:
  - `backend/.../budget/BudgetItemServiceImpl.java`（含 `queryByProject`，可改造为矩阵 DTO）
  - `backend/.../budget/ProjectBudgetController.java`

- **三级边界**:

```
✅ Always（必须做）:
- 金额字段全程使用 BigDecimal（Java）/ DECIMAL(18,2)（MySQL），前端展示两位小数，不使用 float/double
- 提交编制前，Service 层执行全量合规性校验（无红色超限单元格），校验不通过则整体拒绝，不做部分提交
- 矩阵锁定（状态=已编制）后，只读展示，不允许 inline 编辑（前端 disabled + 后端接口拒绝）
- 每次 cell 更新写入操作日志（budget_change_log，action=CELL_EDIT）

⚠️ Ask First（需确认再做）:
- 修改「软提醒/硬限额」阈值（当前 100%/110%），需更新 budget_param 表配置并评估影响
- 增加新预算科目（新增 budget_subject 记录），需确认是否影响历史矩阵已提交行的合规性
- 跨年度复制矩阵（将上年度金额复制为本年度初始值），需评估是否影响已有草稿

🚫 Never（禁止做）:
- 不得使用 float/double 存储或计算金额（精度丢失风险）
- 不得允许「已编制」状态的矩阵直接修改单元格（必须走调剂流程，即 §2.3）
- 不得在前端计算合规性（所有金额校验在 Service 层，防伪造请求绕过）
```

### 2.3 调剂审批（Tab3 — tab-transfer.html）— 对应 Issue #3216 #2259

**功能描述**: 项目经理申请区域间或科目间预算调剂，走审批流（项目经理 → 财务 → 吴耀），审批通过后 `BudgetTransferApprovedEvent` 触发矩阵自动更新。

- **组件**:
  - `TransferApplyForm`（调剂申请表单：来源区域/科目/金额 / 目标区域/科目/金额 / 调剂原因 / 附件上传）
  - `TransferListTable`（调剂申请列表：申请编号 / 调剂方向 / 金额 / 申请人 / 当前审批节点 / 状态 / 操作）
  - `TransferDetailModal`（申请详情弹窗：完整流程节点 + 各节点审批人/时间/意见 + 矩阵变更预览）
  - `TransferApprovePanel`（审批操作面板：通过/驳回/转审，含审批意见输入框，仅当前节点审批人可见）
  - `TransferFlowDiagram`（审批流示意图：可视化流程节点状态，当前节点高亮）

- **数据源**:
  - `POST /wande/budget/transfer`（提交调剂申请）
  - `PUT /wande/budget/transfer/{transferId}/approve`（审批操作：通过/驳回/转审）
  - `GET /wande/budget/transfer`（调剂申请列表，支持 `projectId` / `status` / `applyTime` 筛选）

- **交互**:
  - 「申请调剂」按钮 → 弹出申请表单 → 来源/目标各选区域+科目，系统自动显示来源区域该科目当前可用余额（防止申请超出可用额度）
  - 调剂金额 ≤ 来源区域余额的 20%（可配参数）：仅需财务审批；> 20%：加吴耀审批节点
  - 列表按「待我审批」/ 「我提交的」/ 「全部」三个视角切换
  - 审批通过 → Spring ApplicationEvent `BudgetTransferApprovedEvent` 触发，监听器自动更新矩阵（来源扣减/目标增加），矩阵 Tab2 刷新
  - 驳回申请 → 状态变为「已驳回」，申请人收到系统通知（内部消息 + 企微）

- **代码现状**: 🟡 有基础需改造（使用 Spring ApplicationEvent 解耦，矩阵更新通过事件触发）
  - 调剂申请和审批主体逻辑需新建 `BudgetTransferController` + `BudgetTransferService`
  - 审批节点路由逻辑（20% 阈值判断、多级审批人配置）参考现有审批体系模块，尽量复用 `ApprovalFlowService`
  - `BudgetTransferApprovedEvent` 需新建，`BudgetMatrixService` 中注册监听器方法

- **已有文件**:
  - 审批体系公共服务：`backend/.../approval/ApprovalFlowService.java`（可复用）

- **三级边界**:

```
✅ Always（必须做）:
- 调剂申请提交前，Service 层校验来源余额 ≥ 申请调剂金额，不足则拒绝
- 审批通过后，矩阵更新必须在同一事务内（BudgetTransferApprovedEvent 同步监听），保证一致性
- 所有调剂记录永久保留（不可删除），支持审计追溯
- 调剂审批操作写入 budget_change_log（action=TRANSFER_APPROVE / TRANSFER_REJECT）

⚠️ Ask First（需确认再做）:
- 调整调剂金额阈值（20% 规则），需更新 budget_param 表并评估是否影响进行中的申请
- 增加审批节点（如增加总监审批层），需修改审批流路由逻辑
- 允许跨项目调剂（当前仅支持同项目内区域间调剂）

🚫 Never（禁止做）:
- 不得允许调剂金额超过来源区域余额（Service 层硬拦截）
- 不得在审批通过后直接操作 budget_item 表（必须通过 BudgetTransferApprovedEvent 解耦触发）
- 不得允许申请人自己审批自己的调剂申请（Service 层校验当前审批人 ≠ 申请人）
```

### 2.4 BOM 联动（Tab4 — tab-bom.html）— 对应 Issue #3213 #3219

**功能描述**: 将采购单/BOM 行绑定到对应预算行，绑定后 BOM 清单实时展示剩余可用额度（余额条）。采购员创建采购单时，系统自动校验对应预算行余额，超额拦截。

- **组件**:
  - `BomLinkTable`（BOM 联动清单：BOM 行 ID / 物料名称 / 计划金额 / 绑定预算行 / 已消费 / 余额 / 余额条（进度条形式））
  - `BomBindModal`（绑定预算行弹窗：搜索选择区域+科目 → 显示该科目剩余余额，确认绑定）
  - `BalanceAlertBanner`（余额告警横幅：余额低于 10% 的预算行列表 + 告警颜色标识）
  - `BomSummaryStats`（汇总统计卡：已绑定 BOM 数 / 覆盖率 / 绑定总金额 / 余额合计）
  - `UnboundBomList`（未绑定预算行的 BOM 条目列表，提示运营及时绑定）

- **数据源**:
  - `POST /wande/budget/bom-link`（绑定 BOM 行到预算行）
  - `GET /wande/budget/bom-link`（获取联动清单，含余额条数据）

- **交互**:
  - BOM 清单默认展示当前项目所有 BOM 行（含已绑/未绑），按绑定状态过滤
  - 点击某 BOM 行「绑定预算」→ 弹窗选择区域+科目 → 显示可用余额 → 确认
  - 绑定后，该 BOM 行的「余额条」实时更新，若余额 < 计划金额的 20%，余额条变红
  - 「未绑定」Tab 列出所有尚未关联预算行的 BOM 条目，便于批量处理
  - 点击余额条详情 → 展开消费明细（历史采购单列表，含时间/金额/经手人）

- **代码现状**: 🟡 有基础需改造
  - `backend/.../execution/ProjectBomController.java` 已存在，含 BOM 列表接口，需增加 `budget_item_id` 关联字段
  - 余额计算逻辑需新建：`budget_item.total_amount - SUM(consumed_amount)` 实时聚合
  - 采购单创建时的余额拦截逻辑需在 `PurchaseOrderService` 中新增调用预算余额检查

- **已有文件**:
  - `backend/.../execution/ProjectBomController.java`（BOM 列表已有，需增加预算关联字段）

- **三级边界**:

```
✅ Always（必须做）:
- 采购单创建时，Service 层强制校验对应预算行余额 ≥ 采购金额，余额不足直接拒绝（HTTP 422）
- 余额条进度百分比 = 已消费 / 预算总额，精确到两位小数，前端不做四舍五入
- BOM 绑定关系记录一旦建立，不得单方面解绑（需双方（采购/预算负责人）确认或走调剂流程）
- 余额变更实时推送到已打开 BOM 联动 Tab 的前端（WebSocket / SSE，可选）

⚠️ Ask First（需确认再做）:
- 允许一个 BOM 行绑定多个预算行（按比例拆分），当前版本只支持一对一绑定
- 历史采购单（budget_item 关联前已创建的）的追溯绑定方式
- 余额预警阈值（当前 10%）调整，需更新 budget_param 表

🚫 Never（禁止做）:
- 不得允许采购金额超过绑定预算行余额（Service 层硬拦截，不依赖前端展示余额提示）
- 不得将 BOM 联动清单接口对无预算查询权限的用户开放（查询时需校验 wande:budget:bom:read）
```

## 7. PageGuide 内容

- **这是什么**: 万德项目总控预算管理中心，以「区域 × 预算科目」二维矩阵统一管理项目从设计到执行的全周期预算。

- **解决什么问题**: 替代 Excel 管预算、明道云填报、钉钉审批分散在三个平台的旧模式。核心痛点：设计/编制/执行三阶段数据不打通，超支发现滞后，调剂无记录，保证金台账靠手工维护。

- **快速上手**（3 步）:
  1. **建区域**：进入「区域管理」Tab → 新建区域或从招投标方案 Excel 批量导入区域列表 → 区域 ID 自动与 BOM 和计划打通
  2. **填矩阵**：进入「矩阵编制」Tab → 选择项目和年度 → 在区域×科目单元格填入预算金额（超限变黄/红色预警）→ 确认无误后「提交编制」锁定
  3. **看执行**：进入「执行仪表盘」Tab → 热力图实时展示各区域消费率 → 点击红色单元格下钻明细 → 消费曲线对比计划偏差

- **权限说明**: 矩阵编制需「项目经理」角色；调剂申请超过预算 20% 时会自动转吴总审批；超管可查看全部项目和配置全局参数。

## 附录 B：技术决策记录（ADR）

### ADR-1：区域（zone）作为公共轴

- **背景**: BOM 模块、计划模块、预算模块各自维护了一份「区域」数据，导致同一项目区域数据不一致（命名差异、遗漏区域等）。
- **选项 A**: 各模块各自维护区域表，通过接口同步（事件驱动）
- **选项 B**: 统一 `budget_zone` 表作为唯一区域主数据，BOM 和计划通过 `zone_id` 外键引用
- **决策**: 选方案 B，`budget_zone.id` 成为全局 `zone_id`，BOM 模块的 `bom_items.zone_id` 和计划模块的 `plan_task.zone_id` 均引用此主键（不强制 DB 外键，Service 层逻辑约束）。
- **负责人**: 后端 CC，在 Flyway V20260427_01 运行后通知 BOM 和计划模块联调
- **影响**: BOM 模块 `ProjectBomController` 的区域创建端点改为调用 `/budget/zones` API，不再独立建区域

### ADR-2：三级校验策略

- **背景**: 预算超支管控需要在不同阶段采用不同严格程度的约束，一刀切「硬拦截」会影响设计灵活性。
- **三级定义**:
  - **设计阶段（软提醒）**: 单元格金额超过 `design_limit` → 背景变黄（`SOFT_WARN`），允许继续填写，仅提示
  - **编制阶段（硬限额）**: 单元格金额超过 `design_limit × hard_block_ratio`（默认 110%）→ 背景变红（`HARD_BLOCK`），「提交编制」接口拒绝，返回 422
  - **执行阶段（采购/报销硬拦截）**: 采购单金额超出对应预算行余额 → `PurchaseOrderService` 返回 422，拒绝创建
- **决策**: 三级校验系数（`soft_warn_ratio` / `hard_block_ratio`）存入 `budget_param` 表，超管可调整，无需发版
- **补充**: 软提醒和硬限额仅基于「设计限额 vs 编制金额」；执行拦截基于「预算行余额 vs 采购金额」，两套逻辑独立

### ADR-3：调剂审批走 Spring ApplicationEvent 解耦

- **背景**: 调剂审批通过后，需要同时更新矩阵（`budget_item` 两行金额）+ 发送通知 + 记录审计日志。如果在 `ApprovalService.approve()` 中直接调用三个 Service，耦合度高，后续扩展困难。
- **决策**: 审批通过后发布 `BudgetTransferApprovedEvent`（Spring ApplicationEvent），监听器分别处理矩阵更新、通知、审计日志，彼此解耦。
- **实现约束**: 矩阵更新监听器与审批操作在同一事务内（`@TransactionalEventListener(phase = BEFORE_COMMIT)`），保证数据一致性；通知和审计日志使用 `@Async` 异步监听，不阻塞审批响应。
- **超时保护**: 矩阵更新超过 3s 时降级为异步补偿任务（标记 `transfer.status=APPROVED_PENDING_MATRIX_UPDATE`），补偿任务每 5min 扫描并重试。
